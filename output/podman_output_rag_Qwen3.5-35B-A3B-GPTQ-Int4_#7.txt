[Q12/69] Bagaimana cara membatalkan/menghentikan job yang berstatus PENDING di terminal?
------------------------------------------------------------
Untuk membatalkan atau menghentikan job yang berstatus PENDING di terminal, Anda dapat menggunakan perintah `scancel` diikuti dengan ID job tersebut. Berikut adalah langkah-langkahnya:

1. Lihat status dan ID job Anda terlebih dahulu dengan menjalankan perintah `$ squeue -ul $USER`.
2. Identifikasi ID job yang ingin dibatalkan (misalnya, jika ID job adalah 231).
3. Jalankan perintah pembatalan dengan format `$ scancel [job-ID]`, contohnya: `$ scancel 231`.

Perintah ini akan mengubah status job menjadi CA (CANCELED) yang berarti job dibatalkan oleh user.

[Q22/69] Saya punya file simulasi.ipynb. Bagaimana urutan langkah menjalankannya sebagai batch job di Job Composer EWS menggunakan conda environment saya sendiri?
------------------------------------------------------------
Untuk menjalankan file simulasi.ipynb Anda sebagai batch job di Job Composer EWS menggunakan Conda Environment User, ikuti langkah-langkah berikut sesuai dokumen referensi:

1. Login ke EFIRO Web Service, buka menu Jobs > Job Composer, lalu buat ruang job dengan memilih New Jobs > From Template.
2. Pilih template "Conda Environment User (CPU)" untuk berjalan di compute node CPU atau "Conda Environment User (GPU)" jika memerlukan akselerasi GPU di compute node GPU, kemudian isi Job Name dan klik [Create New Job].
3. Siapkan file komputasi melalui menu [Edit Files] untuk upload file simulasi.ipynb (limit upload 100MB per file; jika lebih gunakan Aplikasi SFTP).
4. Lakukan konversi file Notebook menjadi file Python karena batch job hanya menjalankan file .py dengan perintah berikut di [Open Terminal]:
   $ jupytext --to py simulasi.ipynb
5. Cek kelengkapan package conda environment user yang akan digunakan dengan membuka [Open Terminal], mengaktifkan modul pyload user ($ pyl load [nama-modul]), lalu jalankan pengecekan pada file Python hasil konversi ($ pycheck simulasi.py).
6. Jika output menunjukkan MISSING PACKAGE, instal package yang kurang menggunakan perintah pip install --user sesuai panduan output (contoh: $ pip install --user pyusb); perhatikan bahwa user tidak dapat update package bawaan environment, hanya dapat menginstal package yang belum ada.
7. Isi kolom submit script dengan klik [Open Editor], ikuti petunjuk NOTES dan alur script di dalamnya, lalu klik [Save] untuk menyimpan script.
8. Jalankan job dengan mengklik tombol [Submit].
9. Pantau status dan output job dengan meremfresh halaman Job Composer; status Running memungkinkan pemantauan output di kolom Folder Contents (result-jobID.txt untuk output terminal dan error-jobID.txt untuk log status) serta utilisasi CPU dan memori via Monitoring > Grafana.

[Q30/69] Apa bedanya Effective Core Hour dengan Actual Core Hour di dalam sistem ALELEON?
------------------------------------------------------------
Berikut adalah perbedaan antara Effective Core Hour dan Actual Core Hour berdasarkan sistem ALELEON:

1. **Effective Core Hour Remaining** adalah sisa kuota core hour efektif yang menjadi dasar Slurm menjalankan job, di mana nilainya berubah tergantung status job:
   - Ketika tidak ada job berjalan, Effective Core Hour Remaining sama dengan sisa kuota Actual Core Hour.
   - Ketika ada job berjalan, Effective Core Hour Remaining sama dengan sisa kuota Total Core Hour (yang mencakup Booked Core Hour).

2. **Actual Core Hour Usage** adalah penggunaan core hour sesungguhnya secara *realtime* yang dihitung dari waktu job benar-benar berjalan, dan nilai ini menjadi basis perhitungan biaya penggunaan ALELEON.

Secara ringkas, Effective Core Hour Remaining menunjukkan sisa kuota yang tersedia untuk alokasi atau penggunaan saat ini, sedangkan Actual Core Hour Usage mencatat konsumsi sumber daya yang sudah terjadi secara nyata.

[Q32/69] (Troubleshooting) Saya submit job GROMACS tapi selalu gagal dengan pesan error Invalid syntax. Di script saya menulis #SBATCH --mem= 64 GB. Apa yang salah dari tulisan saya?

Masalah pada penulisan perintah `#SBATCH --mem= 64 GB` Anda terletak pada adanya spasi yang tidak seharusnya di antara tanda sama dengan (`=`) dan nilai, serta pemisahan angka satuan (GB) menjadi terpisah.

1. Penulisan yang salah (seperti kasus Anda):
   #SBATCH --mem= 64 GB
   #SBATCH --ntasks = 32
   #SBATCH --time =01:00:00

2. Penulisan yang benar (harus tanpa spasi tambahan dan satuan menyatu):
   #SBATCH --mem=64GB
   #SBATCH --ntasks=32
   #SBATCH --time=01:00:00

Silakan ubah baris tersebut menjadi `#SBATCH --mem=64GB` tanpa spasi setelah tanda sama dengan dan tanpa spasi sebelum "GB".

[Q33/69] Saya submit 3 batch job berturut-turut. Job 1 pakai 64 CPU. Job 2 pakai 32 CPU. Kenapa saat saya submit Job 3 yang butuh 64 CPU, statusnya malah PENDING dengan tulisan QOSMaxCpuPerUserLimit, padahal node epyc masih banyak yang kosong?
------------------------------------------------------------
Status job Anda yang PENDING dengan alasan `QOSMaxCpuPerUserLimit` terjadi karena total alokasi CPU dari Job 1 dan Job 2 telah melebihi batas Limitasi Fair Usage (LFU) atau sisa limit komputasi per user, sehingga sistem menahan Job 3 meskipun node fisik masih tersedia.

Berdasarkan dokumen referensi, berikut adalah penjelasan dan langkah-langkah yang harus Anda lakukan:

1.  **Penyebab Masalah**
    Status `QOSMaxCpuPerUserLimit` menandakan bahwa alokasi CPU yang diminta oleh job melebihi limit atau sisa limit komputasi [Limitasi Fair Usage (LFU)] milik user Anda. Sistem Slurm akan menahan job tersebut sampai ada alokasi sesuai limit QOS yang tersedia, misalnya ketika Job 1 atau Job 2 selesai berjalan.

2.  **Langkah Cek Awal**
    Anda perlu memverifikasi apakah alokasi job Anda melebihi *Fair Limit* dengan menjalankan tool `sausage` pada menu `sfair`:
    ```bash
    sausage sfair
    ```
    Perhatikan kolom "Fair Limit" dan bandingkan dengan alokasi di submit script Anda (padanan SBATCH `-ntasks` untuk CPU).

3.  **Solusi Jika Alokasi Melebihi Limit**
    Jika hasil pengecekan menunjukkan ada alokasi yang lebih besar dari limit, Anda memiliki dua opsi:
    
    *   **Opsi 1: Batalkan dan Revisi Job**
        Batalkan job yang berlebih, ubah jumlah alokasi CPU di submit script agar sesuai limit, lalu jalankan ulang. Untuk membatalkan job secara manual di terminal, gunakan perintah berikut:
        ```bash
        squeue -lu $USER
        scancel [job-ID]
        ```
        Contoh membatalkan job ID 231:
        ```bash
        scancel 231
        ```
        
    *   **Opsi 2: Ajukan Pembukaan Limitasi Sementara**
        Ajukan permohonan pembukaan limitasi sementara ke tim support admin. Tim admin akan memutuskan permintaan Anda menyesuaikan ketersediaan sumber daya komputasi dan antrian job pengguna lain di ALELEON Supercomputer.

4.  **Catatan Penting**
    Jika tidak ada alokasi yang melebihi limit, maka tidak ada masalah teknis; job hanya mengantri hingga ada alokasi memadai sesuai rentang limit alokasi komputasi user atau menunggu job lain milik user/sekelompok akun institusi selesai.

[Q34/69] (Troubleshooting) Job saya berstatus PD dengan alasan AssocMaxWallDurationPerJobLimit. Di script saya menulis #SBATCH --time=4-00:00:00. Akun saya adalah akun perseorangan biasa. Mengapa tertahan?
------------------------------------------------------------
Job Anda tertahan dengan status **PENDING** dan alasan **AssocMaxWallDurationPerJobLimit** karena durasi waktu yang Anda ajukan melebihi batas maksimal (walltime) yang diperbolehkan untuk akun perseorangan.

Berdasarkan dokumen referensi, berikut adalah rincian batasan waktu (walltime) per jenis akun:
1.  **Akun Perseorangan:** Maksimal **3 hari (72 jam)**.
2.  **Akun Institusi:** Sesuai PKSPIAS.
3.  **Akun Uji Coba:** Maksimal **1 hari (24 jam)**.

Dalam script Anda, Anda menulis `#SBATCH --time=4-00:00:00` yang setara dengan **4 hari**. Karena akun Anda adalah **Akun Perseorangan**, batas maksimalnya hanya **3 hari (72 jam)**, sehingga sistem menahan job tersebut agar tidak melebihi limit LFU (Fair Usage Limit).

Untuk memperbaiki masalah ini, Anda perlu mengubah nilai waktu di dalam submit script menjadi maksimal **3 hari** atau kurang. Format penulisan yang benar sesuai dokumen adalah:
*   `#SBATCH --time=3-00:00:00` (untuk tepat 3 hari)
*   Atau format alternatif `HH:MM:SS`, misalnya `#SBATCH --time=72:00:00`.

[Q35/69] Saya mau menjalankan 10 simulasi FLACS-CFD sekaligus menggunakan fitur Slurm Array. Setiap simulasi butuh 4 core CPU dan 8GB RAM. Di script, apakah saya harus menulis --cpus-per-task=40 atau --cpus-per-task=4?
------------------------------------------------------------
Berdasarkan dokumen referensi, Anda harus menulis `#SBATCH --cpus-per-task=4` pada script submit.

Berikut adalah penjelasan dan langkah-langkah yang benar sesuai dokumen:
1. Parameter `--cpus-per-task` mendefinisikan jumlah core thread CPU yang digunakan oleh **setiap** job array (satu simulasi), bukan total keseluruhan. Dalam contoh dokumen untuk 8 job array dengan masing-masing 2 core, penulisannya adalah `#SBATCH --cpus-per-task=2`.
2. Untuk skenario Anda yang menjalankan 10 simulasi (array) dengan masing-masing 4 core, penulisan di script adalah sebagai berikut:
   - `#SBATCH --array=1-10`
   - `#SBATCH --cpus-per-task=4`
3. Total alokasi CPU akan dihitung secara otomatis oleh sistem menjadi 40 core (10 job * 4 core per job), namun perintah di dalam script tetap harus ditulis per task individual.
4. Demikian pula untuk memori RAM, jika setiap simulasi butuh 8GB, maka tulis `#SBATCH --mem=8GB`, sehingga total alokasi memori menjadi 80GB (10 * 8GB).

[Q37/69] Saya butuh komputasi memori raksasa sebesar 350 GB untuk satu aplikasi yang non-MPI (tidak bisa dibagi ke banyak node). Partisi apa yang WAJIB saya gunakan agar tidak error kehabisan memori?

Untuk menjalankan aplikasi non-MPI yang membutuhkan total alokasi memori 350 GB pada satu node, Anda **WAJIB** menggunakan partisi **epyc-jumbo**.

Berdasarkan spesifikasi dokumen referensi:
1. Partisi **epyc** hanya menyediakan RAM efektif sebesar **240GB per node**, sehingga tidak cukup untuk kebutuhan 350 GB Anda.
2. Partisi **epyc-jumbo** menyediakan RAM efektif sebesar **500GB per node**, yang memenuhi syarat minimal 350 GB tersebut.
3. Karena aplikasi bersifat non-MPI (hanya berjalan di satu node), maka seluruh alokasi memori harus tersedia dalam satu node tersebut, dan perhitungan `#SBATCH --mem` akan diterapkan langsung ke nilai 350 GB atau lebih pada partisi epyc-jumbo ini.

Anda dapat menentukan partisi ini dengan perintah berikut dalam script submit:
`#SBATCH --partition=epyc-jumbo`

[Q39/69] Kenapa saat saya meminta alokasi #SBATCH --ntasks=7, sistem Slurm ALELEON akan membulatkannya menjadi 8 dan saya ditagih biaya untuk 8 core?
------------------------------------------------------------
Sistem Slurm ALELEON membulatkan alokasi core CPU ganjil menjadi genap ke atas demi optimasi performa karena fitur SMT (Simultaneous Multi-Threading) yang aktif di mana 2 thread dalam 1 core tidak boleh dipakai oleh 2 job berbeda.

Berdasarkan dokumen referensi, berikut adalah penjelasan rincinya:
1. ALELEON mengaktifkan SMT pada semua CPU compute node sehingga berjalan 2 thread per core.
2. Konfigurasi Slurm memastikan 2 thread dalam 1 core CPU tidak dipakai oleh 2 job berbeda untuk optimasi performa.
3. Oleh karena itu, jumlah alokasi CPU ganjil akan dibulatkan genap ke atas oleh Slurm, contohnya jumlah 7 menjadi 8, 23 menjadi 24, dst.
4. Aturan ini menetapkan bahwa alokasi minimum core CPU haruslah 2 dan genap.

[Q56/69] Bagaimana cara membuat conda environment di aleleon?

1. Aktifkan modul Anaconda yang sesuai dengan kebutuhan Anda, misalnya versi terbaru 2025.06-1 atau versi stabil 2024.02-1 dengan perintah `$ ml Anaconda3/2025.06-1` atau `$ module load Anaconda3/2024.02-1`.

[Q57/69] bagaimana cara menjalankan jupyter dengan conda environment sendiri?
------------------------------------------------------------
Untuk menjalankan sesi Jupyter menggunakan Conda Environment milik Anda sendiri di ALELEON, berikut adalah langkah-langkah lengkap yang harus dilakukan:

1.  **Persiapan dan Instalasi Package**
    Sebelum memulai sesi, pastikan Anda telah mengaktifkan Conda Environment dan menginstal package yang diperlukan (seperti `jupyter` atau `jupyterlab`) melalui terminal agar tidak memakan waktu saat sesi berjalan.
    *   Buka terminal via menu **Apps > Aleleon Shell Access**.
    *   Muat modul Anaconda dengan perintah:
        `$ module load Anaconda3/2024.02-1`
    *   Aktifkan Conda Environment Anda menggunakan perintah `source activate` (sesuai nama lingkungan Anda):
        `$ source activate [nama-conda-env]`
        *(Jika lupa nama environment, jalankan `$ conda env list`)*.
    *   Instal package Jupyter jika belum ada:
        `(env) $ pip install jupyter jupyterlab`
    *   Setelah selesai, Anda dapat menonaktifkan environment dengan `$ conda deactivate`.

2.  **Login dan Persiapan File**
    *   Login ke **EFIRO Web Service**.
    *   Untuk efisiensi, upload file komputasi Anda terlebih dahulu ke direktori HOME melalui menu **Files > Home Directory** (limit 100MB per file) atau gunakan aplikasi SFTP untuk file lebih besar.

3.  **Mengisi Formulir Sesi Jupyter**
    *   Klik aplikasi **Jupyter** di homepage EFIRO atau buka menu **Interactive Apps > Jupyter**.

[Q66/69] Apa perbedaan antara menjalankan batch job via Job Composer EWS dan via terminal Slurm?
------------------------------------------------------------
Berikut adalah perbedaan utama antara menjalankan batch job melalui **Job Composer EWS** dan **Terminal Slurm** berdasarkan dokumen referensi:

1.  **Antarmuka dan Cara Akses**
    *   **Job Composer EWS**: Dilakukan secara grafis melalui browser dengan login ke [EWS] atau [EFIRO Web Service], lalu membuka menu **Jobs > Job Composer**. User membuat ruang job menggunakan template yang tersedia (misalnya BoltzTraP2, ACPYPE, LAMMPS) dan mengisi formulir serta submit script melalui editor web ([Open Editor]).
    *   **Terminal Slurm**: Dilakukan melalui terminal Linux setelah login via SSH atau menu **Clusters > Aleleon Shell Access** di [EFIRO Web Service]. User harus menyiapkan file komputasi di direktori HOME, membuat file submit script (.sh) secara manual dengan format `#!/bin/bash -l`, dan menjalankannya menggunakan perintah baris.

2.  **Persiapan File dan Template**
    *   **Job Composer EWS**: Menggunakan fitur **New Jobs > From Template** untuk memilih software spesifik (seperti "BoltzTraP2 25.3.1" atau "ACPYPE 2023.10.27"). Upload file dibatasi 100MB per file melalui menu [Edit Files], atau menggunakan [Aplikasi SFTP] jika lebih besar. Untuk file Python Notebook (.ipynb), user wajib mengonversinya menjadi .py menggunakan perintah `$ jupytext --to py` di dalam terminal Job Composer.
    *   **Terminal Slurm**: User tidak menggunakan template otomatis dari menu, melainkan menulis struktur header SBATCH (`#SBATCH`) dan logika program secara manual dalam file `.sh`. Persiapan file dilakukan dengan upload ke HOME Directory (limit 100MB/file) atau via SFTP. Tidak ada langkah konversi notebook otomatis yang disebutkan dalam panduan terminal ini.

3.  **Penulisan Submit Script**
    *   **Job Composer EWS**: User mengklik tombol **[Open Editor]** di dalam antarmuka web untuk mengisi script sesuai petunjuk **NOTES** yang sudah disediakan di dalam editor tersebut.
    *   **Terminal Slurm**: User membuat file script sendiri (contoh: `submit.sh`) dengan menyalin template yang diberikan dalam dokumen (termasuk bagian `ml load`, `pyl load`, dan `mpirun`), memastikan penggunaan shebang `#!/bin/bash -l`.

4.  **Perintah Eksekusi dan Monitoring**
    *   **Job Composer EWS**: Job dijalankan dengan klik tombol **[Submit]** di halaman web. Status dipantau dengan me-refresh halaman Job Composer (status: Not Submitted, Queue, Running, Completed, Failed). Output dilihat di kolom **Folder Contents** (file `result-jobID.txt` dan `error-jobID.txt`). Penghentian job dilakukan dengan klik tombol **[Stop]**.
    *   **Terminal Slurm**: Job dijalankan dengan perintah `$ sbatch [nama-submit-script]`. Status dipantau menggunakan perintah `$ squeue -ul $USER` (kode status: PD untuk Pending, R untuk Run). Output dibaca menggunakan perintah standar Linux seperti `ls`, `cat`, `more`, atau `tail`. Penghentian job dilakukan dengan perintah `$ scancel [job-ID]`.

5.  **Fitur Tambahan pada Terminal**
    *   Pada panduan Terminal Slurm (khusus GMX-MMPBSA), terdapat instruksi eksplisit untuk memuat modul OpenMPI (`ml load openmpi/5.0.8...`) dan menjalankan program dengan `mpirun` secara manual di dalam script, sedangkan pada Job Composer EWS, instruksi pemanggilan modul seringkali sudah terintegrasi dalam template atau petunjuk NOTES editor web.

Berikut adalah langkah lengkap untuk membuat conda environment baru dan modul pyload di ALELEON berdasarkan panduan resmi:

2.  Aktifkan modul Lmod Conda yang diinginkan (misalnya versi terbaru Anaconda3) dengan perintah `$ ml Anaconda3/2025.06-1` atau `$ ml Mamba/23.11.0-0`. *Catatan: Hindari menggunakan versi Anaconda3 2024.06-1 karena adanya laporan bug.*
3.  Buat conda environment baru dengan format perintah `$ conda create -n [nama-env] python=[versi-python]` (untuk Anaconda) atau `$ mamba create -n [nama-env] python=[versi-python]` (untuk Mamba). Pastikan Anda mendefinisikan versi Python secara spesifik untuk isolasi site-packages.
4.  Aktifkan conda environment yang baru dibuat tersebut menggunakan perintah `$ source activate [nama-env]`.
5.  Setelah lingkungan aktif, buat modul pyload untuk aktivasi otomatis di sesi berikutnya dengan menjalankan perintah `(env) $ pyl mod`, lalu ikuti alur script yang muncul dengan memilih menu nomor 1.

[Q68/69] Apa saja status job di squeue dan artinya masing-masing?
------------------------------------------------------------
Berikut adalah daftar status job (ST/STATE) yang ditampilkan oleh perintah `squeue` beserta artinya:

1. PD (PENDING) - Job mengantri dan belum berjalan; jika tertahan dalam waktu lama, silakan lihat bagian Troubleshooting 1.
2. R (RUN) - Job sedang berjalan; user dapat memantau output menggunakan perintah seperti ls, cat, more, atau tail.
3. CG (COMPLETING) - Job telah selesai dan sedang dalam proses pembersihan (clean-up).
4. CA (CANCELED) - Job dibatalkan secara manual oleh user.
5. PR (PREEMPETED) - Job dibatalkan oleh tim admin, dengan notifikasi alasan dikirimkan melalui email support.
6. S (SUSPENDED) - Job ditahan oleh tim admin, dengan notifikasi alasan dikirimkan melalui email support.

[Q69/69] Bagaimana cara mengisi formulir Jupyter di EWS untuk conda env user?
------------------------------------------------------------
Berikut adalah langkah-langkah mengisi formulir Jupyter di EWS untuk Conda Environment User berdasarkan dokumen referensi:

1.  Login ke **EFIRO Web Service** dan buka menu **Interactive Apps > Jupyter**.
2.  Isi kolom **Number of CPU threads** sesuai jumlah core CPU yang dibutuhkan.
3.  Pilih opsi **Use GPU for the job**: pilih **no** jika tanpa GPU, atau **yes** (yang akan mengalokasikan 1x RTX 3090) jika membutuhkan akselerasi GPU.
4.  Isi kolom **Number of hours** sebagai reservasi waktu maksimal sesi Jupyter, yang ditentukan berdasarkan pilihan jumlah CPU, GPU, dan sisa Core Hour yang tersedia.
5.  Pada bagian **Choose package to run**, pilih opsi **User provided**.
6.  Pada bagian **Choose how to launch Jupyter**, pilih **Anaconda / Mamba**.
7.  Pada bagian **Choose Anaconda/Mamba version & User provided Conda/Mamba environment**, masukkan modul Lmod conda dan nama conda env user yang telah dibuat.
    *   Untuk membantu pengisian, Anda dapat membuka menu **Clusters > Aleleon Shell Access**, lalu jalankan perintah `$ pyl info [nama-modul]` untuk melihat informasi modul pyload user Anda.
8.  Klik tombol **[Launch]** untuk memulai sesi.
